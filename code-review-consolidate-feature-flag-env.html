做得好的地方

  • OpenSpec 驅動開發： proposal → design → 分 capability spec → tasks 鏈條完整， 尤其 performance-invariants 把 hot-path 規則寫成 SHALL + 檔案錨點，利於長期維護。
  • internal/flags 設計： EnvEnabled falsy 語意一致、 Gate + ResetForTest 符合 otel-mongo 既有快取契約；四模組檔案一致。
  • otel-nats 三層 gate： natsPropagationGate 在 construction 快取到 tracedConn.propagationEnabled，hot path 不讀 env；inject/extract 正確分離。
  • otel-nats connImpl： public API 乾淨委派，disabled path 無 OTel span； 與 spec「public method 無 tracingEnabled 分支」一致。
  • 測試覆蓋 propagation 矩陣： TestDefaultBehaviorIsTracingOnlyNoHeaders、 TestPropagationFlagRestoresFullWireOutput 等直接鎖定 v0.3→v0.4 行為變更，避免 regression。
  • otel-mongo 既有 strategy-split： Collection/Cursor 等 internal/{direct,traced,shared} 是此 change 的模板；v1/v2 parity 意識在 CLAUDE.md 與 spec 中反覆強調。

必須修正（Critical）

1. Demo 堆疊未啟用 NATS propagation — 端到端 trace 會斷

新預設下，僅 OTEL_NATS_TRACING_ENABLED=true 會產生 local span， 不會 注入 traceparent。目前配置： 

docker-compose.yml / charts/otel-traces-test/values.yaml（api、worker 等）：
  OTEL_INSTRUMENTATION_GO_TRACING_ENABLED=true
  OTEL_NATS_TRACING_ENABLED=true
  # 缺少 OTEL_NATS_PROPAGATION_ENABLED=true

建議： 在所有發布/訂閱 NATS 的服務加上：

OTEL_NATS_PROPAGATION_ENABLED=true

並修正 proposal.md Impact 段落，刪除或改寫「no env wiring changes needed」。

2. Proposal 與實際部署不一致

文件聲稱 consumer 無需變更，但 propagation 預設 OFF 的破壞性變更要求 顯式 env。 在歸檔 OpenSpec change 前應統一敘述，並在 README 提供 grep 遷移食譜（tasks 12.12）。

結論

此 OpenSpec change 的核心價值已實現：統一 default-OFF 語意、共享 flags helper、 NATS propagation 獨立開關，以及 otel-nats 層級的策略分流與豐富的 regression 測試。 然而依 tasks.md 與 capability spec 的嚴格解讀，整體仍屬進行中： 文件、CI 守衛、layout 統一、gorilla-ws/mongo Client 遷移，以及 demo 部署的 propagation env 尚未對齊。建議先修復 NATS propagation 部署缺口並跑通 E2E，再考慮歸檔 OpenSpec change。
